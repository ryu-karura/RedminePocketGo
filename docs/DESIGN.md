# 詳細設計書

本書は、システム全体の構成と設計判断をまとめたものです。
構築の手順は [SETUP.md](SETUP.md)、日常の操作は [MANUAL.md](MANUAL.md) に
分けて記載しています。本書では手順は扱いません。

---

## 1. 目的と前提

### 1.1 目的

既存の Redmine を、スマートフォンから快適に参照・更新できるようにします。
あわせて、パスワードの手入力を日常から排除し、Redmine の API キーが
利用者の端末に残らない構成を実現します。

### 1.2 前提

- Redmine はすでに運用されているか、本プロジェクトの Docker 構成で新規に
  構築される。
- Redmine の REST API が有効化されている。
- 利用者は 1 人で複数の端末（スマートフォンと PC）を併用する。
- 社内ネットワークまたはインターネット越しに HTTPS で公開される。

### 1.3 対象外

- Redmine 本体の改造
- オフライン編集
- プッシュ通知（iOS の PWA 制約が大きいため、初期リリースでは扱わない）

## 2. 全体構成

### 2.1 構成図

```
┌─────────────────────────────┐
│ ブラウザ（スマホ / PC）           │
│  HTML5 SPA                  │
│  ・パスキーの生成と署名（WebAuthn）  │
│  ・API キーは一切保持しない          │
└──────────────┬──────────────┘
               │ HTTPS / 同一オリジン
               │ セッション Cookie（HttpOnly）
               ▼
┌─────────────────────────────┐
│ Go 中継サーバー                  │
│  ・静的ファイル配信                │
│  ・パスキー認証（WebAuthn）        │
│  ・セッション管理                  │
│  ・API キー保管庫（暗号化）          │
│  ・Redmine REST API 中継        │
└──────────────┬──────────────┘
               │ HTTP(S)
               │ X-Redmine-API-Key
               ▼
┌─────────────────────────────┐
│ Redmine + PostgreSQL          │
│  Docker で構築                  │
└─────────────────────────────┘
```

### 2.2 なぜ中継サーバーを置くのか

SPA から Redmine の REST API を直接叩く構成も成立しますが、次の理由で
採用しません。

| 論点 | 直接続 | 中継あり |
|---|---|---|
| 認証方式 | API キーまたは Basic 認証のみ。パスキー不可 | パスキーに対応できる |
| API キーの所在 | ブラウザ内。XSS で漏洩する | サーバー内のみ |
| CORS | Redmine 側で許可設定が必要 | 同一オリジンのため不要 |
| 通信回数 | 画面ごとに複数回 | サーバー側で集約できる |
| 権限の絞り込み | できない | 許可リストで制御できる |

モバイル回線では往復回数が体感速度を大きく左右するため、集約できる点は
実利としても大きいと判断しました。

### 2.3 同一オリジン配信

SPA の静的ファイルと API を同じ Go サーバーから、同じオリジンで配信します。
これにより次が同時に満たされます。

- CORS の設定が不要になる
- `SameSite=Lax` の Cookie がそのまま使える
- パスキーの RP ID とアプリの配信元が一致する

オリジンを分ける構成は、リフレッシュトークンなどの追加設計が必要になるため
採用しません。

## 3. 認証設計

### 3.1 方式

WebAuthn（パスキー）を主たる認証方式とします。Go 側は
`github.com/go-webauthn/webauthn` を使用します。

Discoverable Credential（Resident Key）を要求し、ユーザー名の入力なしに
ログインできるようにします。ユーザー検証（`userVerification`）は
`required` とし、生体認証または PIN を必須とします。

### 3.2 エンドポイント

| メソッド | パス | 内容 |
|---|---|---|
| POST | `/auth/register/begin` | パスキー登録の開始。チャレンジを返す |
| POST | `/auth/register/finish` | 登録の完了。公開鍵を保存する |
| POST | `/auth/login/begin` | 認証の開始。チャレンジを返す |
| POST | `/auth/login/finish` | 認証の完了。セッションを発行する |
| POST | `/auth/logout` | セッションを破棄する |
| GET | `/auth/me` | 現在のセッション情報を返す |

### 3.3 初回登録（ブートストラップ）

パスキーが 1 つも登録されていない状態では、パスキーでログインできません。
最初だけ Redmine の認証情報で本人確認を行い、その場でパスキーを登録します。

```
1. 利用者が Redmine のログイン名とパスワードを入力
2. Go サーバーが Redmine の /my/account.json に Basic 認証で問い合わせ
3. 成功したら、レスポンスに含まれる API キーを取得
4. API キーを暗号化して保存し、ユーザーレコードを作成
5. その場で WebAuthn の登録セレモニーを実行
6. 以降はパスキーのみでログイン可能
```

Redmine のパスワードはこの一連の処理の中でのみ使用し、保存しません。

### 3.4 2 台目以降の端末

すでにパスキーを持つ端末からログインし、設定画面から追加登録します。

```
1. 端末 A（登録済み）でログイン
2. 設定画面で「別の端末を追加」を選択
3. 6 桁の登録コードが発行される（有効期限 10 分、1 回限り）
4. 端末 B でログイン画面の「登録コードで追加」を選択しコードを入力
5. 端末 B で WebAuthn の登録セレモニーを実行
6. 端末 B のパスキーが同じユーザーに紐付く
```

QR コードによるコード受け渡しも同じ仕組みの上に実装します。

### 3.5 セッション

| 項目 | 値 |
|---|---|
| 保持方法 | Cookie（サーバー側にセッションレコード） |
| Cookie 名 | `rmapp_session` |
| 属性 | `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/` |
| 有効期間 | 既定 30 日（設定可能） |
| 更新 | アクセスのたびに最終利用時刻を更新。有効期間の半分を過ぎたら再発行 |
| 失効 | ログアウト、期限切れ、該当パスキーの削除 |

更新系リクエスト（POST / PUT / DELETE）には CSRF トークンを併用します。
トークンはログイン時に返し、SPA が `X-CSRF-Token` ヘッダーで送ります。

## 4. API キーの管理

### 4.1 紐付けの単位

要件として「ユーザーと端末ごとに保存し、紐付け可能なこと」「スマホと PC を
併用し、混在を想定する」が挙げられています。

これを満たす設計として、**API キーはユーザー単位で 1 つ保持し、パスキーは
端末単位で複数保持する**構成を採ります。

```
ユーザー（1）
  ├─ パスキー（N） ← 端末ごと。スマホ、PC、予備端末が混在
  └─ Redmine API キー（1） ← 暗号化して保管
```

理由は次のとおりです。

- Redmine が発行する API キーは、そもそもユーザーごとに 1 つです。端末ごとに
  別の値を持たせることはできません。
- 端末ごとに別レコードを作ると、Redmine 側でキーを再生成したときに、
  すべての端末で個別に再紐付けが必要になります。
- どの端末からアクセスしても、Redmine 上の操作者・更新履歴が同じ人物として
  記録されるべきです。

端末ごとの識別が必要な用途（最終アクセス日時の表示、端末単位の失効）は、
パスキーのレコード側に属性として持たせます。API キーを分ける必要はありません。

### 4.2 端末レコードが持つ情報

| 項目 | 用途 |
|---|---|
| 表示名 | 「iPhone」「会社の PC」など。利用者が編集可能 |
| 種別 | スマートフォン / PC / タブレット（User-Agent から推定、編集可能） |
| 登録日時 | 設定画面での一覧表示 |
| 最終利用日時 | 使われていない端末の把握 |
| AAGUID | 認証器の種類の判別（任意） |

設定画面ではこの一覧を表示し、端末単位で削除できます。削除された端末の
セッションは即座に失効します。

### 4.3 暗号化

- 方式は AES-256-GCM。
- 鍵（KEK）は設定から与えます。値そのもの、またはファイルパスで指定します。
- ノンスはレコードごとに乱数生成し、暗号文と一緒に保存します。
- 平文の API キーは、リクエスト処理中のメモリ上にのみ存在します。
- API キーを保持する型は、JSON 化すると必ず `"[redacted]"` になるよう
  実装します。ログ・エラー・レスポンスへの誤出力を型で防ぎます。

### 4.4 キーが無効になった場合

Redmine 側で API キーが再生成されると、中継したリクエストが 401 を返します。
このとき Go サーバーは次のように振る舞います。

1. 該当ユーザーの API キーを「無効」としてマークする
2. SPA には `code: "redmine_credential_invalid"` を返す
3. SPA は再紐付け画面を表示し、Redmine の認証情報の再入力を求める
4. 再入力に成功したら新しい API キーを保存する

このときパスキーは失効させません。端末の再登録は不要です。

Go サーバーが持つデータベースです。既定は SQLite、必要に応じて PostgreSQL を

進行中のセレモニーの状態を保持します。有効期限は 5 分。期限切れは定期的に
削除します。

## 6. API 設計

### 6.1 方針

SPA が呼ぶ API は、Redmine の REST API をそのまま透過させるのではなく、
画面が必要とする形に整えて返します。透過が適切なものだけを中継します。

### 6.2 中継の許可リスト

`(メソッド, パス)` の組を明示的に列挙し、一致しないリクエストは 404 を
返します。前方一致による一括透過は行いません。

| メソッド | パス | 用途 |
|---|---|---|
| GET | `/projects.json` | プロジェクト一覧 |
| GET | `/projects/{id}.json` | プロジェクト詳細 |
| GET | `/issues.json` | チケット一覧 |
| GET | `/issues/{id}.json` | チケット詳細 |
| PUT | `/issues/{id}.json` | チケット更新 |
| POST | `/issues.json` | チケット作成 |
| GET | `/issue_statuses.json` | ステータス一覧 |
| GET | `/trackers.json` | トラッカー一覧 |
| GET | `/enumerations/issue_priorities.json` | 優先度一覧 |
| GET | `/projects/{id}/memberships.json` | 担当者候補 |
| GET | `/attachments/{id}.json` | 添付情報 |
| GET | `/my/account.json` | 自分の情報 |

管理系（`/users.json` の作成、`/groups.json` など）は列挙しません。

### 6.3 ヘッダーの取り扱い

| ヘッダー | 扱い |
|---|---|
| `X-Redmine-API-Key` | サーバーが付与。受信したら 400 で拒否 |
| `Authorization` | Redmine へ転送しない |
| `Cookie` | Redmine へ転送しない |
| `X-Redmine-Switch-User` | 受信・送信ともに禁止 |

### 6.4 集約エンドポイント

画面の初期表示に必要なデータをまとめて返します。

| メソッド | パス | 内容 |
|---|---|---|
| GET | `/api/projects/tree` | 全プロジェクトを親子構造に整形して返す |
| GET | `/api/projects/{id}/issues/tree` | チケットを親子構造に整形して返す |
| GET | `/api/issues/{id}/detail` | チケット本体、履歴、添付、選択肢を 1 回で返す |
| GET | `/api/meta` | トラッカー、ステータス、優先度をまとめて返す |

ツリー化はサーバー側で行います。Redmine の一覧 API はページングされるため、
サーバーが全ページを取得してから構造化します。

### 6.5 エラー表現

```json
{ "error": { "code": "redmine_credential_invalid", "message": "..." } }
```

| code | HTTP | 意味 |
|---|---|---|
| `unauthenticated` | 401 | セッションがない、または期限切れ |
| `forbidden` | 403 | 権限がない |
| `not_found` | 404 | 対象がない、または許可リスト外 |
| `invalid_request` | 400 | 入力が不正 |
| `redmine_credential_invalid` | 409 | API キーが無効。再紐付けが必要 |
| `upstream_error` | 502 | Redmine 側の障害 |
| `rate_limited` | 429 | 呼び出し過多 |

`message` は開発者・ログ向けです。利用者に見せる文言は SPA が `code` から
決定します。

### 6.6 キャッシュ

| 対象 | 方針 |
|---|---|
| トラッカー・ステータス・優先度 | サーバー側で 10 分キャッシュ |
| プロジェクトツリー | サーバー側で 60 秒キャッシュ（ユーザー単位） |
| チケット一覧・詳細 | キャッシュしない。SPA 側で ETag を利用 |

キャッシュは必ずユーザー単位で分離します。権限の異なる利用者へ他人の
データが漏れないようにするためです。

## 7. 画面設計

### 7.1 デザインの方針

- **モバイルファースト**。幅 360px を基準に設計し、`min-width` の
  メディアクエリで広い画面に拡張します。
- **色は意味を持たせる**。ステータス、優先度、プロジェクトの階層は色で
  区別しますが、色だけに頼らずアイコンまたはラベルを併記します。
- **触れる領域は 44×44px 以上**。片手操作を前提に、主要な操作は画面下部に
  配置します。
- **コントラストは WCAG 2.1 AA**（本文 4.5:1 以上）を満たします。

### 7.2 デザイントークン

色・余白・角丸・書体は CSS カスタムプロパティで一元管理します。
コンポーネントの CSS に直接の色指定や固定ピクセル値を書きません。

#### 基本色

| トークン | ライト | 用途 |
|---|---|---|
| `--color-bg` | `#f7f8fa` | 画面背景 |
| `--color-surface` | `#ffffff` | カード、リスト行 |
| `--color-border` | `#e2e5ea` | 区切り線 |
| `--color-text` | `#1c2430` | 本文 |
| `--color-text-muted` | `#5b6572` | 補助テキスト |
| `--color-primary` | `#1f6feb` | 主要な操作、リンク |
| `--color-primary-contrast` | `#ffffff` | 主要色の上の文字 |

ダークテーマは同じトークン名で別の値を割り当て、`prefers-color-scheme` と
利用者の明示的な選択の両方に対応します。

#### 状態色

| トークン | 色 | 用途 |
|---|---|---|
| `--color-success` | `#1a7f4b` | 完了、成功 |
| `--color-warning` | `#b45309` | 注意、期限間近 |
| `--color-danger` | `#c02b2b` | エラー、期限超過 |
| `--color-info` | `#0e7490` | 情報 |

#### チケットステータスの色

Redmine のステータスは自由に定義できるため、固定の対応表は持ちません。
`is_closed` フラグと表示順から次のように割り当てます。

| 区分 | 色 | 表示 |
|---|---|---|
| 新規（最初のステータス） | `--color-info` | 塗りつぶしバッジ |
| 進行中（中間） | `--color-primary` | 塗りつぶしバッジ |
| 完了（`is_closed`） | `--color-success` | 枠線バッジ、テキストは淡色 |

#### 優先度の色

| 優先度 | 色 | 表示 |
|---|---|---|
| 低 | `--color-text-muted` | 下向き矢印アイコン |
| 通常 | 表示しない | アイコンなし |
| 高 | `--color-warning` | 上向き矢印アイコン |
| 急いで・今すぐ | `--color-danger` | 二重の上向き矢印 |

#### 階層の色

プロジェクトツリーとチケットツリーでは、階層の深さを左端の縦線の色で
示します。深さに応じて `--color-depth-1` から `--color-depth-5` を
循環させ、インデントと併用します。

### 7.3 ログイン画面

```
┌──────────────────────────┐
│                          │
│        ロゴ                │
│                          │
│   ┌──────────────────┐   │
│   │  パスキーでログイン    │   │  ← 主要ボタン、primary 色
│   └──────────────────┘   │
│                          │
│   ──────── または ────────  │
│                          │
│   登録コードで端末を追加        │  ← テキストリンク
│   Redmine の情報でログイン     │  ← テキストリンク
│                          │
└──────────────────────────┘
```

- パスキーが利用できない環境では、パスキーのボタンを無効化し、理由を
  文章で示します。
- 認証中はボタンをローディング表示にし、多重押下を防ぎます。
- 失敗時は入力欄の下にエラーを表示します。トーストは使いません。

### 7.4 プロジェクト一覧

```
┌──────────────────────────┐
│ ≡  プロジェクト        🔍 ⚙ │  ← ヘッダー固定
├──────────────────────────┤
│ ▼ ┃ 基幹システム            │
│   ┃  ▼ ┃ 会計モジュール      │
│   ┃    ┃  ・ 帳票            │
│   ┃  ▶ ┃ 在庫モジュール      │
│ ▶ ┃ 社内インフラ            │
│ ・ ┃ 総務                   │
└──────────────────────────┘
```

- 親子関係をツリーで表示し、`▼` / `▶` で開閉します。開閉状態は端末に
  保存します。
- 左端の縦線（`┃`）が階層の色を示します。
- 行は「プロジェクト名」と「未完了チケット数」の 2 要素で構成します。
- 検索は名前の部分一致で絞り込み、絞り込み中は該当行の祖先を自動展開します。
- 200 行を超える場合は仮想スクロールを適用します。
- 支援技術向けに `role="tree"` / `role="treeitem"` と `aria-expanded`、
  `aria-level` を付与します。

### 7.5 チケット一覧

```
┌──────────────────────────┐
│ ←  会計モジュール       🔍 ⚙ │
├──────────────────────────┤
│ [状態▾] [担当▾] [優先度▾]    │  ← フィルタ行、横スクロール
├──────────────────────────┤
│ ▼ ┃ #1024 帳票出力の刷新     │
│   ┃ ●進行中  ↑高  山田       │
│   ┃  ・ #1031 PDF 出力       │
│   ┃    ○新規  担当なし        │
│ ・ ┃ #1040 締め処理の高速化    │
│   ┃ ✓完了  通常  佐藤        │
└──────────────────────────┘
```

- 親チケットと子チケットの関係をツリーで表示します。
- 1 行は 2 段組で、上段にチケット番号と件名、下段にステータス・優先度・
  担当者を並べます。
- ステータスはバッジ、優先度はアイコンと色、担当者はテキストで示します。
- 完了チケットは既定で折りたたみ、件数だけ表示します。
- 下端に達したら追加読み込みします。
- 右下にチケット作成のフローティングボタンを置きます。

- 上から「件名」「バッジ」「属性」「説明」「添付」「コメント」の順に

## 8. フロントエンド設計

### 8.1 構成

```
frontend/src/
├── app/          起動処理、ルーター、共通プロバイダ
├── pages/        画面ごとのフォルダ。画面専用の部品もここに置く
├── components/   2 画面以上で使う表示部品
├── features/     ドメインロジック（ツリー化、絞り込みなど）
├── api/          サーバーとの通信。fetch はここにだけ書く
├── stores/       UI の状態
├── styles/       トークンとテーマ
└── types/        型定義
```

### 8.2 状態管理

- サーバー由来のデータは TanStack Query が保持します。グローバルストアに
  複製しません。
- 画面の開閉状態、フィルタの選択などの UI 状態のみをストアに置きます。
- ツリーの構築は `features/` の純粋関数で行い、コンポーネント内では
  組み立てません。テストしやすくするためです。

### 8.3 保存してよいもの

| データ | 保存先 | 可否 |
|---|---|---|
| セッション | Cookie（サーバー管理） | 可 |
| API キー | — | 不可。ブラウザに置かない |
| ツリー開閉状態 | localStorage | 可 |
| フィルタの選択 | localStorage | 可 |
| テーマ設定 | localStorage | 可 |
| チケット本文の下書き | IndexedDB | 可（ユーザー切替時に消去） |

## 9. バックエンド設計

### 9.1 パッケージ構成

| パッケージ | 責務 |
|---|---|
| `config` | 設定の読み込みと検証 |
| `auth` | WebAuthn のセレモニー |
| `session` | セッションの発行・検証・破棄 |
| `credential` | API キーの暗号化保管 |
| `proxy` | 中継、許可リスト、ヘッダー制御 |
| `redmine` | 型付きの Redmine クライアント |
| `httpx` | ミドルウェア、エラー表現 |
| `store` | 永続化 |

`cmd/server/main.go` は依存の組み立てと起動のみを行います。

### 9.2 ミドルウェアの順序

```
RequestID → RecoverPanic → AccessLog → Session → CSRF → ハンドラ
```

### 9.3 Redmine への接続

- 接続には専用の `http.Client` を用意し、タイムアウトを設定します。
- 一時的な失敗（接続エラー、502、503）に限り、指数バックオフで最大 2 回
  再試行します。4xx は再試行しません。
- 同時接続数に上限を設け、Redmine への過負荷を防ぎます。

### 9.4 ログ

`log/slog` による構造化ログを出力します。次のものは記録しません。

- リクエストボディ、レスポンスボディ
- Cookie、セッション ID
- API キー、暗号化キー
- WebAuthn のチャレンジと署名

## 10. 設定項目

設定は起動時に一度だけ読み込み、その場で検証します。必須項目が欠けている
場合は、該当するキー名を示して起動を中止します。

優先順位は、コマンドライン引数 > 環境変数 > 設定ファイル > 既定値 です。
環境変数の接頭辞は `RMAPP_` です。

各項目の具体的な設定方法は [SETUP.md](SETUP.md)、運用中の変更手順は
[MANUAL.md](MANUAL.md) を参照してください。

### 10.1 サーバー

| キー | 必須 | 既定値 | 内容 |
|---|---|---|---|
| `server.addr` | | `:8080` | 待ち受けアドレス |
| `server.base_url` | ✓ | | 公開 URL。パスキーの検証に使う |
| `server.read_timeout` | | `15s` | 読み込みタイムアウト |
| `server.write_timeout` | | `30s` | 書き込みタイムアウト |
| `server.static_dir` | | `./public` | SPA の配置先 |
| `server.trusted_proxies` | | 空 | 前段のリバースプロキシの CIDR |

### 10.2 パスキー

| キー | 必須 | 既定値 | 内容 |
|---|---|---|---|
| `webauthn.rp_id` | ✓ | | RP ID。公開ドメイン名 |

## 11. セキュリティ設計

### 11.1 通信

- HTTPS を必須とします。パスキーの動作要件でもあります。
- HSTS を有効にします。
- Content-Security-Policy を設定し、`script-src 'self'` とします。
  インラインスクリプトは使いません。

### 11.2 権限

Redmine 上の権限がそのまま反映されます。中継サーバーは権限を追加も緩和も
しません。利用者が見られないプロジェクトは、Redmine が返さないため
一覧にも現れません。

### 11.3 想定する脅威と対策

| 脅威 | 対策 |
|---|---|
| XSS による API キー窃取 | ブラウザに API キーを置かない。CSP |
| セッション窃取 | HttpOnly + Secure + SameSite。ID はハッシュ保存 |
| CSRF | SameSite=Lax + 更新系での CSRF トークン |
| 中継の悪用 | 許可リストによるパス制限 |
| 総当たり | ログイン・登録コードに回数制限 |
| 端末紛失 | 設定画面から該当端末のパスキーを即時削除 |

### 11.4 端末紛失への備え

パスキーは端末に紐付くため、唯一の端末を失うとログインできなくなります。
これを防ぐため、次を設計上の必須とします。

- 初回登録の完了時に、2 台目の登録を促す画面を必ず表示する
- 回復コード（1 回限り、10 個）を発行し、印刷または保管を促す
- 回復コードによるログインを実装し、成功後は新しいパスキーの登録を強制する
- 上記が使えない場合の最終手段として、Redmine の認証情報による再紐付けを
  残す（`features.password_bootstrap`）

## 12. 将来機能：地図表示

### 12.1 前提

Redmine 側に `redmine_gtt` プラグインが導入されており、プロジェクトおよび
チケットが位置情報（ポイント、線、多角形）を保持していることを前提とします。

### 12.2 データの取得

`redmine_gtt` は GeoJSON 形式のジオメトリを扱います。中継サーバーは
チケット取得時にジオメトリを含めて返し、SPA が地図に描画します。

ジオメトリを持つチケットが多数ある場合、一覧の応答が肥大化します。
そのため、地図表示用の専用エンドポイントを分けます。

| メソッド | パス | 内容 |
|---|---|---|
| GET | `/api/projects/{id}/geometries` | 表示範囲内のジオメトリを返す |
| GET | `/api/issues/{id}/geometry` | 単一チケットのジオメトリ |

表示範囲（バウンディングボックス）とズームレベルを引数に取り、範囲外の
ものは返しません。

### 12.3 描画

- 地図ライブラリは MapLibre GL JS を使用します。
- ポイントはピン、線は太めのライン、多角形は半透明の塗りと輪郭で描画します。
- 色はチケットのステータスと優先度に対応させ、一覧画面の配色と揃えます。
- 重なったポイントはクラスタ表示にまとめます。
- タイルの配信元は設定で指定します。

### 12.4 画面への統合

- チケット一覧に「リスト / 地図」の切り替えを追加します。
- チケット詳細に、ジオメトリを持つ場合のみ地図を表示します。
- 現在地の表示は、利用者の明示的な操作があったときのみ行います。

初期リリースでは実装しません。設計上、ジオメトリを扱うための拡張余地を
残しておくことのみを本書の対象とします。
